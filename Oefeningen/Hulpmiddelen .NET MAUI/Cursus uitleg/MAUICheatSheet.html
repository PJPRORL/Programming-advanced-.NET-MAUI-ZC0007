.NET MAUI Architectuur & Data Cheat Sheet

Een overzicht van de kernconcepten voor het bouwen van een robuuste .NET MAUI-applicatie, met focus op het MVVM-patroon, databeheer en navigatie.

1. Het MVVM Patroon (Model - View - ViewModel)

Dit is het aanbevolen architectuurpatroon voor MAUI. Het scheidt de UI (View) van de logica (ViewModel) en de data (Model).

Model

  • Doel: Bevat de pure data en business logic. Weet niets van de ViewModel of View.
  • Voorbeeld (Model): User.cs
public class User
{
    public int Id { get; set; }
    public string Name { get; set; }
    public string Email { get; set; }
}

View

  • Doel: De UI (XAML). Is "dom" en bevat idealiter geen code-behind (logica).
  • Binding: Gebruikt Data Binding om te praten met de ViewModel.
  • Voorbeeld (View): UserDetailsPage.xaml
<ContentPage ...
     xmlns:local="clr-namespace:MyApp.ViewModels"
     x:DataType="local:UserDetailsViewModel">

    <Label Text="{Binding CurrentUser.Name}" />
    <Button Text="Opslaan" Command="{Binding SaveUserCommand}" />
</ContentPage>

ViewModel

  • Doel: De "lijm" tussen de View en het Model. Beheert de state (toestand) van de UI en voert acties uit.
  • INotifyPropertyChanged: Essentiële interface. De ViewModel moet deze implementeren zodat de UI weet *wanneer* een property is veranderd.
  • ICommand: Wordt gebruikt voor acties (zoals knop-kliks) i.p.v. "Clicked" events. Dit maakt de logica testbaar.
  • Voorbeeld (ViewModel): UserDetailsViewModel.cs
// (Vaak wordt een BaseViewModel of ObservableObject [uit MvvmToolkit] gebruikt)
public class UserDetailsViewModel : INotifyPropertyChanged
{
    private User _currentUser;

    public User CurrentUser
    {
        get => _currentUser;
        set
        {
            _currentUser = value;
            OnPropertyChanged(); // Vertelt de UI dat 'CurrentUser' is gewijzigd
        }
    }

    public ICommand SaveUserCommand { get; }

    public UserDetailsViewModel()
    {
        SaveUserCommand = new Command(async () => await SaveUser());
    }

    async Task SaveUser()
    {
        // Logica om gebruiker op te slaan...
    }

    // Standaard INotifyPropertyChanged implementatie
    public event PropertyChangedEventHandler PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}

CommunityMVVM Toolkit: Een *must-have* NuGet-pakket (CommunityToolkit.Mvvm). Het vereenvoudigt MVVM enorm met [ObservableProperty] en [RelayCommand] attributen.

2. Data Binding

Data Binding is de techniek die de View en de ViewModel synchroniseert.

BindingContext

  • De View moet weten welke ViewModel zijn databron is.
  • Code-behind (meest voorkomend):
public UserDetailsPage(UserDetailsViewModel vm)
{
    InitializeComponent();
    BindingContext = vm; // Hier stel je de databron in
}

Binding Markup Extensie: {Binding ...}

  • {Binding PropertyName}: Bindt aan een public property op de BindingContext.
    • Voorbeeld: Text="{Binding CurrentUser.Name}"
  • Mode (Richting):
    • OneWay (Standaard): ViewModel → View.
    • TwoWay: ViewModel ↔ View. Essentieel voor invoervelden. 
      <Entry Text="{Binding SearchQuery, Mode=TwoWay}" />
    • OneTime: Alleen ingesteld bij het laden.

3. Collections & CollectionView

De CollectionView is de standaard manier om een lijst met data te tonen.

ObservableCollection<T>

  • Gebruik dit type in je ViewModel in plaats van List<T>.
  • Geeft automatisch een signaal aan de UI wanneer items worden toegevoegd of verwijderd.
  • (Het updatet *niet* de properties *binnen* een item, daarvoor moet het item zelf INotifyPropertyChanged implementeren).
  • ViewModel:
public ObservableCollection<User> Users { get; set; } = new();

XAML (View)

  • ItemsSource: De belangrijkste property. Bindt aan je ObservableCollection.
  • ItemTemplate: Definieert hoe *één enkel item* in de lijst eruitziet.
  • DataTemplate: Het "sjabloon" binnen de ItemTemplate.
  • x:DataType (op de DataTemplate): Cruciaal voor performance en compile-time checking.
<CollectionView ItemsSource="{Binding Users}"
                SelectionMode="Single"
                SelectedItem="{Binding SelectedUser, Mode=TwoWay}">
  
    <CollectionView.ItemTemplate>
        <DataTemplate x:DataType="models:User">
            <Grid>
                <!-- Binnen de DataTemplate is de BindingContext één 'User' object -->
                <Label Text="{Binding Name}" />
                <Label Text="{Binding Email}" />
            </Grid>
        </DataTemplate>
    </CollectionView.ItemTemplate>
  
</CollectionView>

4. Repository Patroon & Interfaces

Scheidt je data-ophalen (API, database) van je ViewModels.

Interface

  • Definieert het "contract". Wat *kan* de repository doen?
  • Voorbeeld: IUserRepository.cs
public interface IUserRepository
{
    Task<User> GetUserById(int id);
    Task<List<User>> GetAllUsers();
    Task SaveUser(User user);
}

Repository (Implementatie)

  • Doet het daadwerkelijke werk.
  • Voorbeeld: UserRepository.cs
public class UserRepository : IUserRepository
{
    // (Hier zou je bijv. een HttpClient of SQLiteConnection hebben)
    public async Task<List<User>> GetAllUsers()
    {
        // Logica om data van een API of database op te halen...
        // ...
        return new List<User>(); // Dummy data
    }
    // ... andere methodes ...
}

Gebruik in ViewModel

  • De ViewModel kent alleen de *interface*, niet de concrete implementatie. Dit maakt het flexibel en testbaar.
  • Voorbeeld: UserViewModel.cs
public class UserViewModel
{
    private readonly IUserRepository _userRepository;

    // De repository wordt "geïnjecteerd" (zie Dependency Injection)
    public UserViewModel(IUserRepository userRepository)
    {
        _userRepository = userRepository;
    }

    public async Task LoadUsers()
    {
        var users = await _userRepository.GetAllUsers();
        // ...
    }
}

5. Dependency Injection (DI)

DI is het mechanisme dat alles automatisch aan elkaar koppelt. In plaats van new ViewModel() te doen, *vraag* je erom in een constructor, en de DI-container geeft het je.

Registratie (in MauiProgram.cs)

Je moet de "services" (zoals Views, ViewModels en Repositories) registreren.

public static class MauiProgram
{
    public static MauiApp CreateMauiApp()
    {
        var builder = MauiApp.CreateBuilder();
        builder.UseMauiApp<App>();

        // === HIER BEGINT DI ===

        // Repositories (Interface -> Implementatie)
        // 'Singleton' = er is maar 1 instance voor de hele app
        builder.Services.AddSingleton<IUserRepository, UserRepository>();

        // ViewModels
        // 'Transient' = elke keer een nieuwe instance
        builder.Services.AddTransient<UserDetailsViewModel>();
        builder.Services.AddTransient<UserListViewModel>();

        // Pages (Views)
        builder.Services.AddTransient<UserDetailsPage>();
        builder.Services.AddTransient<UserListPage>();

        // === EINDE DI ===

        return builder.Build();
    }
}

6. Navigatie (Shell)

.NET MAUI Shell is de aanbevolen manier voor app-structuur en navigatie.

AppShell.xaml

Definieert de structuur (Tabs, Flyout menu).

<Shell ...>
    <!-- Een TabBar onderaan het scherm -->
    <TabBar>
        <Tab Title="Home" Icon="home.png">
            <ShellContent
                Title="Home"
                ContentTemplate="{DataTemplate local:HomePage}"
                Route="HomePage" />
        </Tab>
        <Tab Title="Users" Icon="users.png">
            <ShellContent
                Title="Users"
                ContentTemplate="{DataTemplate local:UserListPage}"
                Route="UserListPage" />
        </Tab>
    </TabBar>
</Shell>

Routes Registreren (voor Detail Pagina's)

Pagina's die niet in de TabBar/Flyout staan, moet je registreren. Dit kan in AppShell.xaml.cs:

public AppShell()
{
    InitializeComponent();

    // Registreer de route voor de detailpagina
    Routing.RegisterRoute(nameof(UserDetailsPage), typeof(UserDetailsPage));
}

Navigeren (vanuit een ViewModel)

  • Gebruik de Shell.Current singleton.
  • await Shell.Current.GoToAsync("UserDetailsPage");
  • *Of (beter):* await Shell.Current.GoToAsync(nameof(UserDetailsPage));

Data Meesturen (Navigatie Parameters)

1. Standaard (Query Property)

Stuur data mee in de URL (simpel type).

// Navigeren: Stuur de user ID 123 mee
await Shell.Current.GoToAsync($"{nameof(UserDetailsPage)}?UserId=123");

// Ontvangen (in ViewModel): Gebruik het [QueryProperty] attribuut.
[QueryProperty(nameof(UserId), "UserId")]
public class UserDetailsViewModel : ObservableObject
{
    [ObservableProperty]
    string userId; // Wordt automatisch gevuld met "123"
    // ...
}

2. Complex (Object)

Stuur een heel object mee via een Dictionary.

// Navigeren: Stuur een heel object mee.
var user = new User { Name = "Jan" };
await Shell.Current.GoToAsync(nameof(UserDetailsPage), new Dictionary<string, object>
{
    { "SelectedUser", user } // Key, Value
});

// Ontvangen (in ViewModel):
[QueryProperty(nameof(SelectedUser), "SelectedUser")]
public class UserDetailsViewModel : ObservableObject
{
    [ObservableProperty]
    User selectedUser; // Wordt automatisch gevuld met het 'user' object
}
